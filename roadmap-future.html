4
Production Hardening
~3–4 weeks
Make the application deployable in real environments: Docker packaging, rate limiting, proper secret management, and health/readiness probes. No new user-visible features — this phase makes existing features reliable at scale.
Deployment
🐳
Docker + docker-compose must
Multi-stage Dockerfile. docker-compose with volume-mounted `data/state.db` and env file for secrets. No root user.
🔁
Reverse proxy guide must
Example nginx config with HTTPS termination, `proxy_pass`, and correct SSE headers (`X-Accel-Buffering: no`).
🗄️
Backup + restore CLI should
`groupware-migrator backup` and `restore` commands for `state.db` with integrity verification.
Security
🚦
Rate limiting on auth endpoints must
Throttle `/auth/login` to prevent brute-force. SlowAPI or custom middleware, keyed by IP. Lock account after N failures.
🔑
Password change + reset flow should
`POST /auth/change-password` (requires current password). Admin-triggered reset token or out-of-band link.
🛡️
Security headers middleware should
Add `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy` via FastAPI middleware.
Reliability
❤️
Liveness vs. readiness probes must
Separate `/health/live` (always 200) and `/health/ready` (checks DB connectivity). Kubernetes-compatible.
🔄
Job retry policy should
Optional auto-retry on transient errors (configurable `max_retries` per job). Exponential backoff. Visible in audit events.
💾
Graceful shutdown with drain should
On SIGTERM, stop accepting new jobs, wait up to N seconds for running jobs to finish checkpointing, then shut down cleanly.
5
Scheduling & Automation
~4–5 weeks
Let migrations run automatically on a schedule, notify users of completions/failures, and support automated pipelines via webhooks. The engine is already incremental-sync-capable — this phase puts that capability on autopilot.
Scheduled Jobs
📅
Cron-style job schedules must
Store cron expressions in `scheduled_jobs` SQLite table. Background scheduler thread fires incremental migrations on schedule. UI to create/pause/delete schedules.
🔁
Schedule history + next-run display must
Dashboard shows next scheduled run, last run time, and last run status for each schedule.
⏸️
Pause / resume schedules should
Toggle a schedule off without deleting it. Useful during maintenance windows.
Notifications
🔔
Webhook notifications must
`POST /auth/webhooks` to register URLs. Fire on job completion, job failure, and batch completion. Signed with HMAC-SHA256. Retry on 5xx.
📧
Email notifications should
Optional SMTP config (`SMTP_HOST`, `SMTP_USER`, etc.) for job completion/failure emails to the job owner. Plain-text + HTML template.
💬
Slack / Teams webhook could
Pre-built webhook format for Slack Incoming Webhooks and Teams connectors. Configured as a webhook type in the webhooks table.
Job Queue
⚖️
Job priority levels should
High/normal/low priority queue. Priority set at job creation time. High-priority jobs preempt normal queue slots.
🚫
Job cancellation must
`POST /api/jobs/{job_id}/cancel` — gracefully stops a running job, marks it cancelled (not failed). Already checkpointed items are preserved.
6
Admin Dashboard & Observability
~3–4 weeks
Give administrators first-class tooling to manage users, inspect system health, and understand migration patterns across all tenants. This is about turning raw data into actionable insight.
Admin UI
👥
User management panel must
List users, create/deactivate accounts, reset passwords, view per-user job counts and last activity. Admin-only tab in the dashboard.
📊
System overview dashboard must
Total jobs (last 7/30 days), success rate, items migrated, active jobs, queue depth. Auto-refreshing cards.
🔍
Cross-user job search should
Admins can search/filter jobs by user, status, date range, protocol, and workload from the admin panel.
Metrics & Monitoring
📈
Prometheus metrics endpoint should
`GET /metrics` (Prometheus format). Expose: active jobs, queue depth, jobs/hour, error rate, SQLite size, worker thread counts.
🪵
Structured JSON logging should
Optional `LOG_FORMAT=json` env var switches to JSON log lines with `job_id`, `user_id`, `duration_ms` fields for log aggregation (Loki, Datadog).
🐛
Sentry / error tracking integration could
Optional `SENTRY_DSN` env var. Capture unhandled exceptions in background workers and API handlers with job/user context.
Audit & Compliance
📋
Admin audit log must
Record all admin actions (user creation, deletion, password reset, API key revocation) in a separate `admin_audit_events` table. Exportable CSV.
🗑️
Data retention policies should
Configurable max age for completed jobs, audit events, and batch records. Background cleanup job runs nightly. Prevents SQLite bloat on long-running instances.
7
Enterprise Auth & Multi-tenancy
~6–8 weeks
Support enterprise authentication patterns and true multi-tenant isolation: organizations with their own data spaces, SSO login, and role-based access beyond the current admin/user binary. Requires careful schema migrations and security review.
Authentication
🏢
SAML 2.0 / OIDC SSO should
Support external IdP (Okta, Azure AD, Google Workspace). Users auto-provisioned on first SSO login. Configurable via `SSO_PROVIDER`, `SSO_ENTITY_ID`, etc.
📁
LDAP / Active Directory bind could
Bind LDAP for credential validation. Group membership maps to roles. Useful for on-premises enterprise deployments.
📱
TOTP two-factor auth should
Optional per-user TOTP (Google Authenticator compatible). QR code enrollment via `/auth/totp/setup`. Recovery codes generated at setup.
Multi-tenancy
🏗️
Organizations / workspaces should
Add `organizations` table. Users belong to one org. Jobs and batches scoped to org. Org admins manage their members. Super-admins see all orgs.
🎭
Role-based access control (RBAC) should
Roles: `viewer` (read-only), `operator` (create/resume jobs), `admin` (user management), `super_admin` (cross-org). Replace boolean `is_admin`.
🔒
Credential vault could
Encrypt connection passwords at rest using AES-256-GCM with a `VAULT_KEY` env var. Decrypt only at connector creation time in worker threads.
Ecosystem
📦
Plugin / connector SDK could
Document the `SourceConnector` / `DestinationConnector` abstract interfaces. Publish a cookiecutter template for custom protocol adapters (e.g., Exchange EWS, Nextcloud).
🌐
REST API versioning should
Introduce `/api/v1/` prefix with explicit API version in responses. Add OpenAPI schema export at `/api/openapi.json`. SDK generation from schema.

Guiding Principles for Future Development

Local-first by default
SQLite stays the default. PostgreSQL or other engines are opt-in via an abstraction layer, not the baseline. New features must work without external services.
Backward compatibility
Existing job payloads, CLI configs, and CSV formats must keep working. Schema migrations are always additive (no column drops, no renames without aliases).
Test coverage gates every phase
Each phase must ship with tests covering the new behavior. No new feature lands without its test. Integration tests hit real SQLite, not mocks.
Security is not an afterthought
New auth features (TOTP, SSO, vault) come from requirements, not retrofits. Rate limiting, audit logs, and RBAC are Phase 4 and 7, not "later".
Simple beats clever
No message queues, no microservices, no ORM until the SQLite + threading model demonstrably breaks. Each layer is added when the layer below it is insufficient.
Deploy anywhere
A single Docker container with one volume mount should be the happy path for every feature. Self-hosted, no cloud dependencies, no license server.

Remaining Gaps — Not Yet Implemented

These items were planned in the roadmap but require external services or a separate integration phase. They are intentionally deferred, not forgotten.

SAML 2.0 / OIDC SSO — Requires selecting and configuring a specific IdP (Okta, Auth0, Keycloak, etc.). Infrastructure groundwork exists; IdP integration is the missing piece.
LDAP / Active Directory bind — Needs a live LDAP server and org-specific schema decisions (attribute mapping, group sync).
Email notifications (SMTP) — Requires an SMTP server. Webhook notifications cover the same use case; email can be added once SMTP config is wired in.
Plugin / connector SDK — A documented extension point for third-party connectors. The connector base classes are ready; what's missing is documentation and a packaging story.
