Groupware Migrator — Product Roadmap Design

Option 1: Foundation → Auth → Frontend. Three phases + backlog. Review each section and let me know in the terminal if anything needs adjusting.

Phase 1 · ~2 weeks · Backend Foundation

Clean the engine room before adding features

🔧 Split app.py into routers

640-line God-file → routers/jobs.py, routers/batches.py, routers/providers.py. Each router owns its domain. Prerequisite for adding auth middleware cleanly.

✅ Pydantic request models

Replace raw dict payloads on all endpoints with typed Pydantic models. FastAPI will auto-validate, auto-document, and surface errors before they reach engine code.

💥 Crash recovery

On startup, any job with status running gets marked failed. Currently these jobs hang forever after a server restart — silent data loss for a product.

🪵 Structured logging

Add Python logging throughout engine and connectors. Configurable log level via env var. Essential for debugging production migrations.

🧹 Remove duplicate code

_batch_status_from_counts is copy-pasted verbatim in both app.py and state.py. Consolidate in state.py, remove from app layer.

⚡ Fix deprecated lifecycle hook

@app.on_event("shutdown") is deprecated in modern FastAPI. Replace with @asynccontextmanager lifespan function.

Phase 2 · ~2–3 weeks · Auth & Multi-tenancy

Make it a real product with user isolation

👤 User model

New users table: id, email, password_hash, is_admin, created_at. Bcrypt hashing. First user bootstrapped from env var or first-run setup page.

🔐 Session management

JWT tokens in HttpOnly cookies. Login/logout endpoints. Auth middleware that protects all /api/* routes. Public routes: /health, /login.

🏠 Per-user job scoping

Add user_id FK to jobs and batches tables. All list/get/stream endpoints filter by the authenticated user. Admins can see all jobs.

🖥️ Login UI

Minimal /login page. On auth success, redirect to dashboard. On session expiry, redirect back to login. No frameworks — matches the existing vanilla JS style.

🔑 API key support

Optional: generate per-user API keys for programmatic access (CLI, scripts). Bearer token auth alongside cookie sessions. Admin UI for key revocation.

Phase 3 · ~1–2 weeks · Frontend Quality

A product-grade UI that doesn't lose your work

📦 ES modules

Split 1231-line app.js into focused modules: api.js, form.js, jobs.js, batches.js, streams.js. No bundler needed — native browser modules.

💾 Form state persistence

Save host/port/username/workload/protocol/provider fields to localStorage on change. Restore on page load. Passwords intentionally excluded.

⏳ Loading states

Disable action buttons and show a spinner during API calls. Prevents double-submissions. Currently nothing stops you clicking "Start Background Job" twice.

📱 Mobile responsiveness

The two-column layout breaks below ~900px. Add responsive breakpoints so the form stacks cleanly on tablet/mobile — important if customers use it themselves.

📝 Copy/description fixes

Hero still says "IMAP/POP3 migration to IMAP" — update to reflect CalDAV/CardDAV support. Small thing, big first impression.

Backlog · Future releases

Valuable but not blocking the product

  • Docker image + docker-compose.yml
  • GitHub Actions CI (lint + tests)
  • OpenAPI schema export & versioning
  • Adaptive SSE (back off when idle)
  • Tasks / Notes workload
  • WebDAV library for CalDAV/CardDAV
  • Webhooks on job completion
  • Rate limiting on batch preflight
  • Role-based access (admin / operator / viewer)
  • Password reset flow

Total estimated effort: ~6–7 weeks for Phases 1–3 (solo developer). Phase 1 is the prerequisite for everything else — don't skip it.
