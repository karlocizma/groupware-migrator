What auth model fits your product?

This determines architecture complexity and how much of the roadmap is auth-related work.

A

Single admin password

One shared secret (env var) gates the entire UI. Simple, fast to build. Good if you operate migrations for customers — customers never log in themselves.

Pros

  • 1–2 days to implement
  • No DB schema changes

Cons

  • No per-user audit trail
  • Shared secret = no revocation
B

Multi-user accounts (email + password)

Each operator gets their own login. Jobs are scoped per user. Fits a team of migration engineers, or customers who self-serve their own migrations.

Pros

  • Per-user audit trail
  • Role-based access possible later

Cons

  • 1–2 weeks to build properly
  • Password reset, session management
C

SSO / OAuth login (Google, Microsoft…)

Customers log in with their existing identity provider. Zero password management on your side. Best for enterprise-facing products.

Pros

  • No credential storage risk
  • Enterprise-ready from day one

Cons

  • 3–4 weeks + IdP setup
  • Overkill if you operate for customers
