1
Backend Foundation
✓ Shipped
~2 weeks
Cleaned up the engine room before adding user-facing features: split the monolithic app.py into focused routers, added typed Pydantic models on all endpoints, and hardened startup/shutdown behavior.
Code Structure
✂️
Split app.py into FastAPI routers
640-line God-file split into routers/jobs.py, routers/batches.py, routers/providers.py. create_app() became a thin wiring function.
🔷
Pydantic request models
Typed JobRequest, BatchRequest, BatchPreflightRequest on all POST endpoints. FastAPI auto-validates and auto-documents.
🧹
Remove duplicated helpers
_batch_status_from_counts was copy-pasted verbatim in both app.py and state.py. Removed the app-layer copy.
Reliability
💥
Crash recovery for in-flight jobs
On startup, any job stuck at status = 'running' is marked failed. Previously these hung forever after a server restart.
📝
Structured logging
Python logging throughout engine/runner.py, engine/preflight.py, and all connectors. LOG_LEVEL env var (default INFO).
🔁
FastAPI lifespan (replace deprecated event hooks)
Replaced @app.on_event("shutdown") with an @asynccontextmanager lifespan. Crash recovery runs on startup inside lifespan.
2
Auth & Multi-tenancy
✓ Shipped
~2–3 weeks
Made the application safe for multiple users: user accounts with bcrypt passwords, JWT sessions in HttpOnly cookies, per-user job scoping, and API keys for CLI access.
Identity
👤
User model & admin bootstrap
users table with id, email, password_hash (bcrypt), is_admin, created_at. First admin bootstrapped from ADMIN_EMAIL/ADMIN_PASSWORD env vars.
🔐
JWT session management
JWT in HttpOnly; SameSite=Strict cookies. POST /auth/login, POST /auth/logout, GET /auth/me. All /api/* routes require auth via fastapi.Depends.
🗝️
API key support
Per-user API keys stored SHA-256-hashed in api_keys table. Bearer token auth alongside cookie sessions. Revocable from the UI.
Scoping & UI
🔒
Per-user job & batch scoping
user_id column on jobs and batches. Non-admin users see only their own. Admins see everything.
🖥️
Login UI page
Dark-glass /login page matching the dashboard style. On 401 API responses → redirect to /login.

Split app.js into ES modules
1231-line monolith → js/api.js, js/form.js, js/streams.js, js/jobs.js, js/batches.js, js/main.js. Native browser modules, no bundler.

4
Production Hardening
✓ Shipped
~3–4 weeks
Made the application deployable in real environments: Docker packaging, rate limiting, proper secret management, and health/readiness probes. No new user-visible features — this phase makes existing features reliable at scale.
Deployment
🐳
Docker + docker-compose must
Multi-stage Dockerfile. docker-compose with volume-mounted data/state.db and env file for secrets. No root user.
🔁
Reverse proxy guide must
Example nginx config with HTTPS termination, proxy_pass, and correct SSE headers (X-Accel-Buffering: no).
🗄️
Backup + restore CLI should
groupware-migrator backup and restore commands for state.db with integrity verification.
Security
🚦
Rate limiting on auth endpoints must
Throttle /auth/login to prevent brute-force. Custom middleware keyed by IP.
🔑
Password change flow should
POST /auth/change-password requires current password. Admin-triggered reset via the admin panel.
🛡️
Security headers middleware should
X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Content-Security-Policy via FastAPI middleware.
Reliability
❤️
Liveness vs. readiness probes must
Separate /health/live (always 200) and /health/ready (checks DB connectivity). Kubernetes-compatible.
🔄
Job retry policy should
Optional auto-retry on transient errors (configurable max_retries). Exponential backoff. Visible in audit events.
💾
Graceful shutdown with drain should
On SIGTERM, stop accepting new jobs, wait up to N seconds for running jobs to finish checkpointing, then shut down cleanly.
5
Scheduling & Automation
✓ Shipped
~4–5 weeks
Let migrations run automatically on a schedule, notify users of completions/failures, and support automated pipelines via webhooks. The engine is already incremental-sync-capable — this phase puts that capability on autopilot.
Scheduled Jobs
📅
Cron-style job schedules must
Cron expressions and interval strings (6h, 30m) in scheduled_jobs table. Background SchedulerThread fires incremental migrations. UI to create/pause/delete.
⏸️
Pause / resume schedules should
Toggle a schedule inactive without deleting it. Next-run and last-run status shown in the dashboard.
Notifications
🔔
Webhook notifications must
Register callback URLs via POST /api/webhooks. Fires on job.completed, job.failed, job.cancelled. HMAC-SHA256 signed. 3-attempt retry on 5xx.
📧
Email notifications ✓ shipped
SMTP-based HTML emails on job completion, failure, or cancellation. Per-user opt-in toggles. Inline-styled dark-theme template with stats card. Admin SMTP test endpoint.
Job Queue
⚖️
Job priority levels should
High / normal / low priority stored per job. Field is present; execution-order enforcement pending ThreadPoolExecutor replacement.
🚫
Job cancellation must
POST /api/jobs/{job_id}/cancel — gracefully stops a running job, marks it cancelled. Checkpointed items are preserved.

7
Enterprise Auth & Multi-tenancy
✓ Shipped
~6–8 weeks
Enterprise authentication patterns and true multi-tenant isolation: organizations with their own data spaces, TOTP 2FA, and role-based access control beyond the original admin/user binary.
Authentication
📱
TOTP two-factor auth should
Optional per-user TOTP (Google Authenticator compatible). QR code enrollment via /auth/totp/setup. 10 recovery codes generated at setup.
🏢
SAML 2.0 / OIDC SSO should (deferred — IdP required)
Infrastructure groundwork exists. Awaiting IdP selection (Okta, Auth0, Keycloak, etc.) and org-specific configuration.
📁
LDAP / Active Directory bind shipped
Bind LDAP for credential validation. Auto-provisions users on first login. Coexists with built-in password auth; set LDAP_HOST to enable.
Multi-tenancy
🏗️
Organizations / workspaces should
organizations + org_memberships tables. Users belong to orgs. Org admins manage their members. Super-admins see all orgs.
🎭
Role-based access control (RBAC) should
Roles: viewer, operator, admin, super_admin. Role rank enforced via fastapi.Depends on every protected endpoint.
🔒
Credential vault could
Fernet encryption for scheduled job credentials at rest via VAULT_KEY env var. Decrypted only at connector creation time in worker threads.
🌐
REST API versioning should
All endpoints accessible at both /api/* and /api/v1/*. OpenAPI schema auto-generated by FastAPI at /openapi.json.
Ecosystem
📦
Plugin / connector SDK ✓ shipped
Abstract SourceConnector / DestinationConnector interfaces published as groupware_migrator.sdk. Third-party connectors register new protocols via setuptools entry points. See examples/sample_plugin/.

12
Scale & Resilience
✅ Complete
June 2026
For deployments that have grown beyond a single-process SQLite instance. PostgreSQL as an opt-in backend, a Redis-backed job queue for horizontal scale, and admin data export — without breaking any single-node deployments.
Storage & Scaling
🐘
PostgreSQL backend
Opt-in via DATABASE_URL. PostgresStateStore inherits from SQLiteStateStore; SQL translated at call sites (?→%s, INSERT OR IGNORE→ON CONFLICT DO NOTHING). SQLite remains the default.
⚖️
Horizontal scaling via Redis job queue
RedisJobManager drop-in for BackgroundJobManager. Jobs pushed to a Redis LIST; workers pop and execute. Cancellation via Redis key polled every 2 s. Start workers with groupware-migrator-worker --redis-url redis://…. Install: pip install "groupware-migrator[redis]".
💾
Admin data export
GET /admin/backup/download — WAL-checkpointed SQLite file download (returns 501 for PostgreSQL backend; use pg_dump). GET /admin/export — full JSON state dump covering all jobs, users, batches, audit events, and OIDC providers.

Architecture Decisions

SequenceFoundation first — auth on top of well-structured code is easier to maintain long-term
Auth modelMulti-user email + password (not SSO — avoids IdP dependency for early product)
Session storageHttpOnly cookies, not localStorage — prevents XSS token theft
No bundlerES modules are sufficient given the project's vanilla JS philosophy
SQLite staysAppropriate for self-hosted SaaS at this scale; no migration to a heavier engine needed
Threading modelThreadPoolExecutor for background jobs; no external queue until SQLite + threading demonstrably breaks

Guiding Principles for Future Development

Local-first by default
SQLite stays the default. PostgreSQL is opt-in via an abstraction layer. New features must work without external services.
Backward compatibility
Existing job payloads, CLI configs, and CSV formats must keep working. Schema migrations are always additive — no column drops or renames without aliases.
Tests gate every phase
Each phase ships with tests covering new behavior. Integration tests hit real SQLite, not mocks — a lesson learned from past mock/prod divergence.
Security is not an afterthought
Rate limiting, audit logs, and RBAC shipped in Phases 4 and 7, not "later". Auth features come from requirements, not retrofits.
Simple beats clever
No message queues, no microservices, no ORM until the SQLite + threading model demonstrably breaks. Each layer is added when the layer below it is insufficient.
Deploy anywhere
A single Docker container with one volume mount is the happy path. Self-hosted, no cloud dependencies, no license server.

Mobile apps
Billing / subscription tiers
